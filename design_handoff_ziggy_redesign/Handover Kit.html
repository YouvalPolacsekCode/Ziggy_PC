Handover kit · v1 · FROM DESIGN → ENG / NEXT DESIGNER

The redesign is a starting line,
not a finished spec.

The new Ziggy direction is excellent — the IA is brutal, the visual system holds up in both palettes, RTL parity works, the dial-first control vocabulary is opinionated and consistent. Keep it. But before anyone ships it, the rest of the iceberg has to be drawn. Ziggy today has 21 routes, 13+ modals, 7 wizards, 8 device-domain controllers, two admin consoles, and roughly 15 settings sub-sections — and the current artboards cover a fraction of it.

The non-negotiable

I want to see and have any and every capability I currently have.

Beyond what was formally agreed in the IA pruning discussion (Scenes killed, HomeMap deferred, QuickAsks/Memory absorbed into Settings, Anomalies absorbed into Alerts, Routines & Suggestions absorbed into Automations, Cameras absorbed into Rooms + Alerts), nothing in the current app may be missing when the redesign ships.

That includes every modal, every wizard step, every settings toggle, every per-domain device controller, the entire admin console, role-gated views, error states, empty states, loading skeletons, and Hebrew/RTL parity for all of it. If it exists today, it has to exist tomorrow — redrawn in the new system, but functionally at parity or better.

06

Cross-cutting concerns

The system layer. None of this is visible as a screen, but it touches every screen.
  • PalettesTwo palettes: data-palette="light" (warm) and data-palette="dark" (premium). Every screen must work in both — not just the seven that have pair artboards today.
  • RTL parityHebrew is a first-class language. Today the redesign mirrors the Dashboard. Every other screen needs the same audit — directional icons (back/forward arrows), bottom-nav order, sliders, breadcrumbs.
  • Roles & gatingFour roles: guest, user, admin, super_admin. Bottom nav, sidebar, and Settings tabs change. Empty / "you don't have access" states needed for each restricted surface.
  • Auth flowsLocal login · invite redemption · password change · session expiry (silent re-login modal, no full reload). Designed today: zero. Today's app dispatches a ziggy:unauthorized event from any 401 to swap in the login UI without dropping the WebSocket.
  • Live state (WebSocket)Every device tile streams state changes in real time. Need explicit "syncing" / "disconnected" affordances. Also: physical IR remote detection (a press on the user's TV remote updates the on-screen state) — toast confirmation pattern.
  • Push notifications (PWA)Service worker registration, VAPID subscription, permission-prompt UX. Today: silent fallback if denied. Owed: a clear "enable notifications" CTA somewhere.
  • Presence pingingLive geolocation watch, 2-min keep-alive, visibility re-pings. Owed: permission prompt UX, "presence is off" banner in the presence widget.
  • ToastsSuccess / info / warning / error. Auto-dismiss timings (3s / 7s for errors). Action toasts ("undo"). Stacking rules.
  • Execution resultsAutomation / routine runs report back: "Evening — 5 steps completed" or "2/5 steps failed: lamp.bedroom unreachable". Needs a per-result expand to see step list.
  • Empty statesEvery list. No rooms, no devices, no automations, no alerts, no tasks, no memory, no quick asks, no cameras, no virtual devices. Each needs its own illustration / copy / primary CTA.
  • Loading skeletonsFirst load, cold-start re-fetch, slow-network. Per-surface skeleton shapes match the eventual content silhouette.
  • Error statesNetwork down, HA disconnected, backend 5xx, entity not found, action failed. Each surface needs a recovery affordance.
  • OnboardingFirst-run trio is drawn. Owed: the actual flow from invite-acceptance to "ready to use" — voice permission, location permission, push permission, household member invites.
07

Definition of done

Before this redesign replaces the current app, every line on this list has to be true. No exceptions.
Every route in §02 either has an artboard or has an entry in §01's pruning ledger.
Nothing falls through the cracks. If a designer can't find a route on either list, it's a bug in this kit.
Every modal and wizard in §03 has at least one drawn state.
20 dialog surfaces. Two are drawn. 18 are owed.
Every wizard has its full step flow drawn, including back, validation, error, and success.
A "Next" button that goes nowhere is not a wizard.
All 8 device-domain controllers in §04 are designed in the new system.
Light, Climate, IR are done. Media Player, Cover, Fan, Lock, Vacuum, Generic are owed.
Every section in §05 exists somewhere in the new Settings.
15 sub-sections. Zero are drawn. Plan the IA inside Settings before drawing the leaves.
Every primary screen has light + dark + RTL variants.
Today seven do. The other ~30 don't.
Every list has its empty state, loading skeleton, and error state.
~20 lists. Each gets three states minimum.
Role-gated surfaces have a "you don't have access" treatment, not a 404.
Today's app silently redirects. Owed: a designed denial.
Every destructive action has a confirm modal with consequences spelled out.
"Delete room — 6 devices will be unassigned" is the bar. Not "Are you sure?".
Tablet wall display and Desktop dashboard cover the same primary surfaces as phone.
Not every modal needs a tablet variant, but Dashboard, Rooms, Room detail, Automations, Alerts do.
Onboarding flow connects start-to-finish, including permission prompts.
Voice, location, push. Each prompt has a "skip" path and the consequences are made clear.
Toast and execution-result patterns are specified once and reused.
Including the multi-line "2/5 steps failed" treatment.
Live state affordances: syncing dot, disconnected banner, physical-remote toast.
The app feels alive today because of these. The redesign has to inherit that feeling.
Sign-off on each pruning decision in §01 with the product owner.
If anyone wants Scenes back, or HomeMap shipped, find out now — not after build.
08

Open questions · before the next round

Decisions I need from you before I keep drawing.
  1. 01
    Is the /ops admin console in scope for the redesign?
    If yes — I'll re-skin Debug Console and Cloud Admin in the new tokens. If no — I'll leave them as a developer surface and we just keep the breadcrumb chrome consistent. Recommendation: out of scope. They're 1% of usage and pixel-fidelity work there is a poor ROI.
  2. 02
    Confirm HomeMap stays deferred — including the feature flag?
    If the flag is still toggleable in admin settings, a super-admin can enable a half-built screen. Cleaner: hide the flag entirely until HomeMap is in scope.
  3. 03
    Virtual Devices — truly admin-only, or surfaced for power users?
    Today it's a full nav item. The pruning agreement buries it under Settings › Admin › Capabilities. Confirm — or do we want a "developer mode" entry point in user-facing nav for the technical user persona?
  4. 04
    Wizard pattern — modal-over-page, full-screen takeover, or inline step expansion?
    Current app uses modal-over-page for everything. New system might benefit from a full-screen takeover for 4+ step wizards (IR, Pairing, Automation). Pick the rule before I draw seven of them.
  5. 05
    Floating voice orb — killed by “Ask” tab, or do both coexist?
    IA doc says the chat tab replaces the orb. But the orb is "always-on, anywhere" — a discoverability advantage. Confirm we're really losing the floating action.
  6. 06
    Bottom nav has 5 slots. We agreed Home · Rooms · Ask · Devices · Automations.
    That means Alerts is one tap deeper (sidebar / overflow). For a unified-alerts pitch that absorbs anomalies + offline + motion, that depth might be wrong. Reconsider, or stand pat?

One last thing

The current Ziggy is sprawling because it earned every feature. Pattern learning, IR detection, family presence, capability authoring, the debug console — none of these are dead weight; they're what makes Ziggy yours. The redesign tightens the front door, not the house. Treat this kit as the floor plan: every room that exists today has to exist tomorrow, redrawn in the new style, no exceptions outside §01.
